## 一、项目概况

TEco Lab 是一个沉浸式 AI 英语学习平台，以母语者原生语料为核心，提供精听、跟读、口语练习等功能。项目采用前后端分离架构，前端 React + TypeScript + Vite，后端 Express + MySQL。

### 当前完成度：约 60-70%

**能用的部分：** 用户注册登录、语料库浏览、视频播放、听写练习、生词本、笔记、收藏、个人中心

**半成品/不能用：** AI 跟读评分（随机分数）、AI 口语陪练（假回复）、签到系统、学习统计

## 四、各模块详细分析

### 1. 用户认证（状态：✅ 基本可用）

**后端：** auth.js — 注册、登录、验证码、重置密码
- 用 bcryptjs 加密密码，JWT 做 token（7天有效期）
- 验证码存在内存 Map 中，5分钟清理一次
- 开发模式下直接把验证码返回给前端

**前端：** Login.tsx + Register.tsx
- 登录需要手机号/邮箱 + 密码
- 注册需要手机号 + 邮箱 + 密码 + 验证码

**主要问题：**
- JWT_SECRET 有硬编码降级 `'your-secret-key-change-this-in-production'`
- 验证码没有真正的短信/邮件发送，只是打印到控制台

### 2. 语料库（状态：✅ 基本可用）

**后端：** corpus.js — 列表、详情、搜索
- 支持按场景/难度/口音筛选，按最新/最热/收藏排序
- 详情返回语料信息 + 所有句子（含时间轴、词汇）
- 访问时增加 view_count

**前端：** CorpusList.tsx + CorpusDetail.tsx
- 列表页：卡片式布局，封面图 + 标题 + 难度星星 + 时长
- 详情页：视频播放器 + 字幕切换 + 变速 + 循环 + A-B循环 + 句子列表 + 词汇弹窗
- 字幕支持6种模式：无/英文/双语/音标/逐句/挖空

**亮点：** 播放器功能完善，字幕模式丰富，体验不错
**主要问题：** 缺少真实语料数据，需要 seed 脚本

### 3. 精听学习（状态：✅ 核心逻辑可用）

**后端：** learning.js — 学习记录增删查、进度追踪、统计

**前端：** Learning.tsx
- 精听模式：视频播放 + 听写输入 + 自动比对评分
- 跟读模式：4种跟读模式 + 录音 + 假评分结果
- 听写评分是客户端做的，按单词比对计算正确率

**亮点：** 听写功能扎实，单词比对逻辑正确
**主要问题：** 跟读模式的评分是假数据

### 4. 跟读 Shadowing（状态：❌ 模拟数据）

**后端：** shadowing.js
- 创建记录：随机生成发音/语调/流利度/重音分数（85 + Math.random() * 10）
- 返回的 feedback 是固定模板，提到 'hello' 的发音纠错
- 查询和统计功能是真实数据库查询

**主要问题：** 完全没有真正的语音分析，分数无意义

### 5. AI 口语陪练（状态：❌ 模拟数据）

**后端：** speaking.js
- 对话存储在内存 Map 中
- AI 回复从5个固定模板里随机选
- 纠错/建议/评分全是写死的
- 结束对话后生成固定报告（语法90/词汇85/流利88/总分87.67）

**前端：** Speaking.tsx
- 完整的聊天 UI：场景选择、对话气泡、打字指示器、录音按钮
- 复盘报告页面：评分卡片、亮点、改进建议、核心表达、对话记录
- 所有交互都是前端 setTimeout 模拟

**主要问题：** 整个功能是空壳子，要接真实 AI 才有意义

### 6. 生词本（状态：⚠️ 后端完整，前端基础）

**后端：** vocabulary.js
- 添加生词时用固定模板数据（ipa/释义/例句都是预设的）
- 支持按场景/掌握度筛选
- 间隔复习算法：1-3-7-14-30 天递增

**前端：** UserCenter 里的生词本 tab
- 展示生词列表，显示掌握度星星
- CorpusDetail 有词汇弹窗和"加入生词本"按钮

### 7. 笔记/收藏（状态：⚠️ 后端完整，前端基础）

**后端：** notes.js + favorites.js
- 完整的 CRUD，关联语料和句子
- 收藏会更新 corpus 表的 favorite_count

**前端：** UserCenter 相应 tab，但功能较基础

### 8. 签到/统计（状态：⚠️ 后端有，前端半成品）

**后端：** checkIn.js + learning.js 的 statistics 接口
- 签到记录 + 连续天数计算
- 按日/周/月聚合学习数据

**前端：** UserCenter 的打卡按钮 + 概览 tab

## 五、核心问题总结

### 1. 范围过大（最主要问题）

1680 行的 PRD 定义了 11 张表、10 个路由模块、5 个学习场景、会员系统。对于一个 solo 初学者项目，这个范围至少大了 3 倍。

结果是：每个功能都做了一半，没有一个是打磨完成的。

### 2. 过度设计

- **过度设计的数据库适配层**：抽象基类 + SQLite 适配器 + MySQL 适配器 + SQL 转换器 + JSON 处理器 + 错误映射器，整整 7 个文件。新项目用 MySQL 就直接用 mysql2，一个 `db.js` 搞定连接，不要搞适配器模式
- **i18next**：用户是中文用户，不需要国际化
- **Antd**：引入了大型 UI 库但没用几个组件，TailwindCSS 已经完全够用
- **会员/支付字段**：数据库里有 `is_premium` 和 `premium_expires_at`，但永远不会用到
- **Redux Toolkit**：对于这个规模的项目，用 React Context + useState 完全够，Redux 增加了大量样板代码

### 3. Mock 数据没有意义

跟读评分和口语陪练是整个项目最吸引人的功能点，但它们的评分数据是随机生成的，反馈是写死的模板。

**问题不在于"做了 mock"，而在于"用 mock 搭建了完整的 UI"。** 两个功能的前端页面加起来超过 1000 行代码（完整的聊天界面、评分卡片、复盘报告），但背后都是假数据。

### 4. 代码层面的小问题

- JWT_SECRET 硬编码降级
- 验证码纯内存存储，重启丢失
- 前端 .tsx 文件大量使用 `any` 类型，TypeScript 的优势没发挥
- 一些后端路由用了 `db.query()` 有的用 `db.prepare().all()`，写法不一致
- 脚本文件散落在 `scripts/` 和根目录，混乱

## 八、对新项目的具体建议

### 数据库结构（精简版）

只需要 5 张表起步：

```sql
-- 1. 用户
CREATE TABLE users (
  id INT AUTO_INCREMENT PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  nickname VARCHAR(100),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 2. 语料
CREATE TABLE corpus (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(200) NOT NULL,
  description TEXT,
  scenario VARCHAR(50),
  difficulty INT,
  video_url VARCHAR(255),
  thumbnail_url VARCHAR(255),
  duration_seconds INT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 3. 句子（关联语料）
CREATE TABLE sentences (
  id INT AUTO_INCREMENT PRIMARY KEY,
  corpus_id INT NOT NULL,
  sentence_index INT NOT NULL,
  start_time DECIMAL(10,3) NOT NULL,
  end_time DECIMAL(10,3) NOT NULL,
  english_text TEXT NOT NULL,
  chinese_text TEXT,
  FOREIGN KEY (corpus_id) REFERENCES corpus(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 4. 学习记录
CREATE TABLE learning_records (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  corpus_id INT NOT NULL,
  sentence_id INT,
  duration_seconds INT,
  accuracy DECIMAL(5,2),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (corpus_id) REFERENCES corpus(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 5. 生词本
CREATE TABLE vocabulary (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT NOT NULL,
  word VARCHAR(100) NOT NULL,
  definition TEXT,
  sentence_id INT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**db.js 示例（MySQL + mysql2）：**
```javascript
import mysql from 'mysql2/promise'

const pool = mysql.createPool({
  host: process.env.DB_HOST || 'localhost',
  user: process.env.DB_USER || 'root',
  password: process.env.DB_PASSWORD || '',
  database: process.env.DB_NAME || 'teco_lab',
  waitForConnections: true,
  connectionLimit: 10,
})

export async function query(sql, params = []) {
  const [rows] = await pool.query(sql, params)
  return rows
}

export async function get(sql, params = []) {
  const [rows] = await pool.query(sql, params)
  return rows[0] || null
}

export async function run(sql, params = []) {
  const [result] = await pool.query(sql, params)
  return { insertId: result.insertId, affectedRows: result.affectedRows }
}
```

**核心思路：** 不要用"未来可能需要"来决定现在的架构。用"现在最少需要什么"来决定。

### 一些具体的教训

1. **先做核心闭环，再想扩展功能。** 一个能用的精听功能 > 五个半成品功能
2. **Mock 可以，但别为 mock 搭完整 UI。** 先验证 API 可行，再做界面
3. **不要为了"学习技术"而引入复杂工具。** 用最少的工具完成功能
4. **PRD 不要写超过 200 行。** 写多了就会忍不住想做，做了就做不完
5. **TypeScript 的 any 是敌人。** 既然用了 TypeScript，就别用 any 逃避
6. **数据库设计越简单越好。** 5 张表能做的事，不需要 11 张表
7. **一个文件能搞定就不要拆。** 比如数据库初始化，一个 `db.js` 用 mysql2 就够了，不要搞适配器模式
